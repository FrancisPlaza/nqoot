Design Analysis

  • Overview

    • Purpose and Goals

      The goal of the system is to provide a forum-like platform for internet users to create posts or post questions ("post") and comment on existing posts ("comment"). The basic implementation of this system will allow users to have several posts and other users will be able to comment on posts as well as give comments on existing comments. Users will also be able to upvote (and downvote) comments. An additional feature ("advanced version") is to allow forum staff to endorse posts and/or comments. In our use case, endorsed posts and comments will appear on top of other posts and/or comments in the website.

    • Unusual Requirements

      One of the most essential requirements of this system is auto-updating of the page as comments come in. In this case, users do not need to refresh the page everytime in order to see new comments. It is essential for the page to get or receive new information from the server. That is, the implementation approach can have a page that constantly queries the server for new information (i.e. comments) or the server has an open channel between all its clients and send new information as they come in. The latter approach might be harder to implement.

    • Key Features

      The crowd-approved comment system has the following features:

      • Allow users to create posts and add comments to posts as well as existing comments
      • Upvote (and downvote) comments to existing posts; also, upvote on posts
      • For the advanced version, allow staff to endorse posts as well as comments

    • Event Model

      User events

      • add_post: user creates new post
      • edit_post: user edits post that belongs to it
      • delete_post: user removes post that belongs to it
      • up_vote: user upvotes a post or an existing comment to a post
      • down_vote: user downvotes an existing comment to a post
      • add_comment: user adds comment to an existing post

      For the advanced version, a staff user exists. The staff inherits all user events.

      Staff events

      • endorse_post: staff endorses and existing post
      • endorse_comment: staff endorses existing comment
      • delete_post: staff deletes any post
      • delete_comment: staff deletes any comment

      Standard scenario

      • user_login
      • add_post
      • edit_post
      • add_comment
      • up_vote
      • down_vote
      • user_logout

      Orderings

      • User ::= add_post edit_post* delete_post* add_comment* up_vote* down_vote*
      • Staff ::= endorse_post* endorse_comment* delete_post* delete_comment*

  • Behavior

    • Feature descriptions

      • Allow users to add, edit, and delete posts as well as comment on existing posts.
      • Enable staff to monitor posts including endorsing and moderating posts and comments.

    • Security concerns

      While there isn't a huge security risk in this platform or system, it is necessary to ensure the credibility of the user. That is, users can only upvote and downvote posts that doesn't belong to it. It must not also have access to staff previliges. It follows that only staff accounts can only have access to staff privileges.
